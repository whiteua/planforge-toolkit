# Error Taxonomy

## Levels

| Level | Use when |
|---|---|
| `critical` | План нельзя считать реализованным: отсутствует ключевой контракт, есть data-loss/security риск, workflow ломается полностью. |
| `high` | Значимая функциональность работает неверно или неполно, но есть обходной путь. |
| `medium` | Локальный дефект, edge case, неполный тест/валидация, который не ломает основной сценарий. |
| `low` | Косметика отчёта, слабая формулировка, поддерживаемость без прямого риска выполнения. |

## Categories

- `logic` — неверная бизнес-логика, edge cases, неверные переходы статусов.
- `contract` — CLI/API/schema/format не совпадает с планом.
- `code` — программный дефект, типы, lifecycle, обработка ошибок.
- `tests` — отсутствует обязательная проверка или тестовый сценарий.
- `security` — path traversal, injection, auth/validation риск.
- `perf` — неоправданная сложность, неприемлемый scope без stop condition.
- `ops` — запуск/окружение/инструменты не соответствуют плану.
- `db` — миграции/схема/данные, если план затрагивает БД.
- `math` — формулы, расчёты, округление.
- `code-plan-mismatch` — реализация есть, но не того поведения, которое описано в MASTER_PLAN.

## Issue Quality

Каждая новая ошибка должна содержать:

- конкретный affected task/phase;
- evidence с файлами/строками или честный `not found`;
- почему это реально нарушает план;
- action-oriented recommendation;
- уровень severity, не завышенный формально.
