# Status Rules

## Task Statuses

| Marker | Canonical | Meaning | Final-blocking |
|---|---|---|---|
| `✅` | `verified` / `fully-implemented` | Контракт реализован полностью | no |
| `⚠️` | `partial` / `implemented-with-errors` | Реализовано частично или с дефектами | yes |
| `⭕️` | `missing` | Реализации нет | yes |
| `‼️` | `deviates` | Реализация противоречит плану | yes |
| `not-applicable` | `not-applicable` | Пояснительный пункт без реализации | no |

`ambiguous` считается незакрывающим состоянием и обычно отображается как `⚠️` с пояснением:
кодовый путь найден, но состояние реализации нельзя уверенно классифицировать. Такая строка
обязана содержать `evidence_anchor`. Если тот же task id повторяет `⚠️ ambiguous` с тем же
`evidence_anchor`, что и в предыдущем отчёте, это только предупреждение валидатора; само
правило повтора не повышается до exit 1.

## Issue Statuses

| Marker | Canonical | Meaning |
|---|---|---|
| `✅` | `fixed` | Ошибка устранена |
| `⚠️` | `fixed-with-errors` | Устранена частично или с побочными дефектами |
| `⭕️` | `not-fixed` | Не исправлена |
| `⚠️` | `regressed` | Была закрыта, но снова проявилась |

## Propagation

- Если у задачи есть open issue level `critical` или `high`, задача не может быть `✅`.
- Если evidence отсутствует или противоречив, задача не может быть `✅` без явного объяснения.
- `not-applicable` не участвует в расчёте закрытия плана.
- Final report допустим только когда все задачи `✅`/`not-applicable`, а все issue-блоки `✅ fixed`.

## Self-Consistency & Ambiguous

Для блокирующих вердиктов (`✅` на critical-задаче, `⭕️`, `‼️`) агент один раз пере-выводит
вердикт из того же evidence. Расхождение понижает задачу до `⚠️ ambiguous` и требует
`evidence_anchor`. Перед следующим проходом агент усиливает или заменяет evidence, иначе тот же
якорь снова даст тот же спорный результат.

Если та же задача остаётся `⚠️ ambiguous` с тем же `evidence_anchor`, что и в предыдущем отчёте,
валидатор выдаёт предупреждение об осцилляции. Это warning-only проверка: она не превращает
промежуточный отчёт в ошибку валидации и не меняет общий смысл `⚠️` как незакрывающего статуса.

Неразрешённую ссылку плана можно фиксировать как `AMBIGUOUS_REFERENCE`: агент не сканирует весь
план в поиске нескольких похожих invocation matches, а явно отмечает неоднозначность и
возвращается к ближайшему проверяемому якорю.

Залипание в открытом состоянии эскалируется через `consecutive_open_invocations` / `ask_required`.
`consecutive_open_invocations` считает только повторение той же ошибки с тем же fingerprint.
Если реализация изменилась и issue получает новый fingerprint, цепочка считается новой и счётчик
начинается заново.

## Evidence Strengthening

Паттерны усиления evidence при расхождении:

- добавить ready read-only probe (тест/type-check) и записать exit-code;
- расширить якорь с одной строки до блока функции (`path#Lx-Ly` → весь метод);
- добавить cross-reference на тестовый файл, покрывающий контракт.

Закрытая задача (`✅`) с составным id (`<phase>.<task>`) обязана иметь `evidence_anchor`.
В финальном отчёте отсутствие якоря даёт exit 1; в промежуточном отчёте это предупреждение.
Фаза без точки в id считается свёрткой дочерних задач и отдельного якоря не требует.

`confidence: low` не может закрывать задачи финального отчёта и issue со статусом `fixed`.
В финальном отчёте такой закрывающий вердикт даёт exit 1; в промежуточном отчёте это
предупреждение. `confidence: high` и `confidence: medium` сохраняют обычный смысл статуса.
