# Pass First

Первый проход используется, когда `bootstrap.pass_type == first` и прошлых отчётов нет.

## 1. Read Master Plan

Прочитать MASTER_PLAN полностью. Блоки кода и цитаты не считать actionable-задачами, но можно
использовать как контекст для понимания контрактов.

## 2. Enumerate Phases And Tasks

Основные эвристики:

- `##`/`###` с `Фаза 1`, `Phase 1`, `Stage 1`, `1.`, `1)` становятся фазами;
- вложенные списки и подзаголовки становятся задачами с ID `<phase>.<sub>`;
- plain text план без Markdown превращается в фазу `1` со строковыми задачами;
- сохранить номера строк, чтобы evidence был проверяемым.
- при классификации задач завершённая задача с dotted-id (`<phase>.<sub>`) обязана иметь
	`evidence_anchor`; отсутствие якоря в промежуточном отчёте является предупреждением
	валидатора, а не основанием финально закрывать задачу.

## 3. Trace Code

Для каждой actionable-задачи извлечь токены:

- имена файлов и директорий;
- классы, функции, методы, enum/constant names;
- API endpoints, route names, table names, migration names;
- явно указанные config keys/env vars.

Искать через Read/Grep/Glob. Не запускать mutating commands. Каждый результат получает
`Traceability`:

- `verified` — код реализует контракт;
- `partial` — часть контракта есть, часть отсутствует;
- `missing` — следов нет;
- `deviates` — реализация есть, но противоречит плану;
- `ambiguous` — несколько конфликтующих мест или evidence недостаточен;
- `not-applicable` — пункт не является задачей реализации.

Если задача реализована частично или состояние реализации нельзя уверенно классифицировать,
использовать `⚠️ ambiguous` с evidence и не закрывать её как done. Если готовой read-only
пробы для проверки нет, пометить пункт `probe-needed`; агент не генерирует новую пробу и не
запускает нераспознанные команды.

## 4. Detect Issues

Создавать issue только на реальные риски реализации:

- отсутствующий или неверный контракт;
- нарушенная бизнес-логика, формула, edge case;
- неверная интеграция между модулями;
- security/auth/validation пробел;
- performance/ops риск, который явно заявлен в плане;
- падение разрешённой read-only пробы.

## 5. Resolve Statuses

Статус задачи зависит от худшего связанного результата. Critical/high ошибка не позволяет задаче
быть `✅`. `ambiguous` не закрывает задачу.

Первый проход не должен финально закрывать low-confidence задачи или issue. Если evidence слабый,
неполный или отсутствует `evidence_anchor` для закрываемой dotted-id задачи, статус остаётся
незакрывающим до следующего проверяемого evidence.

## 6. Write Report

Сформировать отчёт по [REPORT-FORMAT.md](REPORT-FORMAT.md). `## Задача` должна быть достаточной
для внешнего исполнителя: что исправить, где искать, как проверить.

Warnings от валидатора переносить в warning-секцию отчёта, чтобы следующий проход видел все
предупреждения без повторного запуска валидатора.
