# Pass Subsequent

Повторный проход используется, когда `bootstrap.pass_type == subsequent`.

## A. Re-audit Known Issues

1. Распарсить LAST_REPORT через `parse-report`.
2. Для каждой carry-over ошибки проверить evidence в текущем коде.
3. Назначить новый статус:
   - `✅ fixed` — симптом исчез и регрессий не видно;
   - `⚠️ fixed-with-errors` — исправление частичное или породило побочный дефект;
   - `⭕️ not-fixed` — проблема сохраняется;
   - `regressed` — было закрыто, но снова актуально; в отчёте отображать как `⚠️ regressed`.
4. Если старая ошибка не содержит evidence, пометить её `ambiguous-prev`, оставить прежний статус и
   потребовать уточнить локацию.

## Evidence Anchor Drift

`evidence`-якоря `path#Lx-Ly` из прошлых отчётов — **advisory**. Между проходами файл могли
править, поэтому перед переиспользованием якоря агент сверяет код по указанным строкам с текущим
состоянием. При сдвиге/несовпадении якорь считается невалидным, а вердикт пере-выводится заново
(не наследуется автоматически).

## B. Full Master Plan Re-scan

1. Повторить Pass First steps 1-4 поверх текущего кода.
2. Сравнить новые findings с carry-over списком:
   - исчезнувшие старые ошибки остаются в отчёте как `✅ fixed`;
   - сохраняющиеся старые ошибки остаются carry-over;
   - новые ошибки добавляются после carry-over;
   - регрессии переоткрываются.
3. Стабильно перенумеровать issue-блоки от 1.
4. Пересчитать статусы фаз/задач.
5. Выбрать open/final имя по общему правилу: final только при пустом наборе открытых ошибок и
   всех задачах `✅`/`not-applicable`.

## Master Plan Drift Between Invocations

Если SHA текущего MASTER_PLAN отличается от `master_plan_sha256` в LAST_REPORT, продолжить аудит,
но добавить предупреждение в `## Дополнительный контекст`: мастер-план изменился между проходами,
поэтому статусы пересчитаны по текущему содержимому.
