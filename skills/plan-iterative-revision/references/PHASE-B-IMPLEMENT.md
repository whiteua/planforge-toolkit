# PHASE B — IMPLEMENT

Цель фазы: применить **все** контракты из только что созданного `REVIEW(N)` к главному плану.
Никакие другие файлы не трогаются — ни review (включая текущий), ни код.

## Inputs

- `PLAN`       — путь к главному плану.
- `REVIEW(N)`  — только что созданный review-файл (выход Фазы А).

## Output

- Обновлённый `PLAN`. SHA-256 файла должен отличаться от исходного.

## Step 1 — Load & sort

1. Выполнить машинную проверку review-файла:

  ```bash
  python "<SKILL_DIR>/scripts/next_review_index.py" validate-review "<REVIEW(N)>"
  ```

  Если validator вернул ошибку — **не начинать Фазу Б**. Вернуться к Фазе А и исправить
  только что созданный review до валидного состояния.
2. `Read REVIEW(N)` целиком.
3. Распарсить issues. Для каждого взять: `id`, `severity`, `location_in_plan`, `required_fix`,
   `acceptance`.
4. Отсортировать:
   - Primary key:  `severity` в порядке `blocker → major → minor → nit`.
   - Secondary:    позиция `location_in_plan` в плане сверху вниз (стабильная).
5. Зафиксировать стартовый SHA-256 главного плана (для финальной проверки в WORKFLOW).

## Step 2 — Apply one issue at a time

Для каждого issue из отсортированного списка:

```
attempts = 0
loop:
    Read PLAN (целиком или нужный участок)        # обязательный re-read
    locate target by location_in_plan
    construct old_str (с 3-5 строками контекста до и после)
    construct new_str по required_fix
    apply Edit (одна транзакция, один issue)

    if Edit failed (old_str не уникален / не найден):     # stale old_str
      Read целевой секции плана заново → пере-извлечь актуальный old_str
        attempts += 1
        if attempts >= 2:
            mark issue as deferred-conflict (см. ниже)
            break
      continue          # один реактивный повтор с обновлённым old_str

    Read изменённый участок                        # verify
    проверить, что изменение соответствует required_fix
    проверить acceptance (текстовая проверка: ключевые фразы / структуры на месте)
    if not OK:
        attempts += 1
        if attempts >= 2:
            mark issue as deferred-conflict
            break
        continue

    issue done; goto next
```

### Жёсткие правила

- **Один issue = одна транзакция Edit.** Никаких батчей разных issues одной правкой.
- **Re-read обязателен перед каждым Edit** — даже если только что что-то меняли. Это гасит
  stale-context bugs.
- **Контекст в `old_str`** — минимум 3 строки до и 3 после, чтобы матч был уникален.
- **Запрет ложной замены.** Если искомый фрагмент невозможно найти даже после свежего `Read`,
  немедленно откажись от правки и пометь `deferred-conflict`. НЕ подбирай «похожий» `old_str` —
  ложная замена не того фрагмента опаснее отложенного issue. Следующая Фаза А выявит расхождение.
- **Не модифицировать `REVIEW(N)`**. Если по ходу работы выяснилось, что `required_fix`
  неоднозначен, **не править ревью**. Применить максимально близкое к контракту изменение
  и пометить issue как `deferred-conflict` с заметкой; следующая Фаза А выявит расхождение.
- **Не создавать новые файлы.** Если контракт требует создать файл — отразить это в плане
  (текстом), но физически файл создаёт человек/другой скилл.
- **Не править код.** Если контракт требует изменить код — отразить как пункт плана.

## Step 3 — Deferred-conflict handling

Если issue не применился за 2 попытки:

1. Не создавать новый файл, не править ревью.
2. Запомнить in-memory: `{issue_id, reason, last_error}`.
3. После завершения цикла по всем issues — вывести в stdout (для логов) краткий отчёт:
   `"Deferred N issues: [id1, id2, ...] — будут переоткрыты в следующей Фазе А."`
4. WORKFLOW передаёт управление обратно к Фазе А; невыполненный контракт автоматически
   обнаружится через contract verification (PHASE-A-AUDIT.md, Step 1) и попадёт в `REVIEW(N+1)`
   с повышенной severity.
5. **Счётчик подряд-отложений (C3, дополняет авто-повышение severity, не дублирует его):**
   - При каждой пометке issue как `deferred-conflict` инкрементировать `deferral_count[fingerprint]`
     (fingerprint берётся из issue; см. REVIEW-FILE-FORMAT.md).
   - При успешном применении issue в этой или будущей Фазе Б — сбросить/удалить счётчик.
   - Если `deferral_count[fingerprint] >= K_STUCK` (K_STUCK = 3) — issue считается ЗАСТРЯВШИМ:
     пробросить сигнал WORKFLOW (`stuck=[fingerprint]`). WORKFLOW останавливает цикл и пишет
     финальный review с `**Status**: ESCALATED` (см. WORKFLOW «Stop conditions» и REVIEW-FILE-FORMAT).
   - Severity-cap: счётчик НЕ повышает severity сверх `blocker`; вечного «звона» на blocker нет.

## Step 4 — Final self-check

После применения всех issues:

1. `Read PLAN` целиком.
2. Для каждого **успешно применённого** issue:
   - Проверить `acceptance` ещё раз на свежем содержимом плана.
   - Если не подтверждается — допатчить (одна попытка). При неуспехе — пометить
     `deferred-conflict` и добавить в отчёт.
3. Сравнить новый SHA-256 со стартовым. Если совпадают — **ошибка**: ничего не применилось.
   Сообщить пользователю, не передавать управление в WORKFLOW (выйти аварийно).
4. Выполнить diff guard (если проект под git): между snapshot-ами до/после Фазы Б не должно
  появиться новых изменённых файлов, кроме `PLAN`.

## Forbidden in Phase B

- Любые `Edit` / `Write` в любые файлы, кроме `PLAN`.
- Создание новых файлов любого типа.
- Любые правки `REVIEW(N)` или предыдущих ревью.
- Любые правки кода проекта.
- Запуск тестов, сборок, линтеров.
- Изменения в нескольких issues одной транзакцией Edit.

## Done criteria for Phase B

- Все non-deferred issues применены и прошли acceptance check.
- Deferred issues задокументированы для следующей итерации.
- SHA-256 главного плана изменился.
- Diff guard подтверждает, что Фаза Б изменила только `PLAN` (или пользователь получил явное
  сообщение, почему git snapshot недоступен).
- Управление возвращено в WORKFLOW.
